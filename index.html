Why

Software security ends where firmware begins

UEFI replaced BIOS with more capability — and a much larger attack surface. Once malware lives in SPI flash, it survives OS reinstalls and disk swaps, and it can blind or feed false data to anything running above it.

The gap

  • Rootkits and bootkits persist in SPI flash, surviving wipes and reinstalls
  • Antivirus runs at the OS layer — useless if firmware is already compromised
  • Secure Boot and TPM still depend on the firmware they're meant to verify
  • Forensic tools running from inside a compromised system can be fed false results

The approach

  • SOIC-8 clip gives direct, non-invasive access to the SPI flash chip
  • STM32 reads the firmware image straight off the chip, before host CPU boot
  • STM32 calculates a SHA-256 hash and compares it against a trusted hash
  • On mismatch, BootShield automatically restores clean firmware from a signed backup

System

The verification flow

Every stage from clip attach to "windows open for us" — exactly as it runs on the board.

Step 01 · Interface

SOIC-8 clip → SPI flash chip

The clip connects directly to the motherboard's SPI flash chip — the memory chip that stores the firmware first executed when the computer powers on. No soldering required.

SOIC-8 test clip
Step 02 · Control unit

STM32 reads & hashes firmware

STM32 is the brain of the system. It reads the firmware image off the flash chip, calculates its SHA-256 hash, and compares it against a stored trusted hash.

STM32F103C8T6 · SHA-256
Step 03 · Decision

Hash match?

Match → green LED, boot is allowed. Mismatch → red LED, BootShield restores clean firmware and re-checks the hash before letting the host proceed.

Green = allow · Red = restore
